Character device driver

Ressources utiles : 0. The Linux Kernel Archives 1. The Linux Kernel documentation 2. The Linux Kernel Module Programming Guide 3. Linux Kernel Labs

Sous UNIX, l'accès aux périphériques matériels s'effectue via de fichiers situés dans le répertoire /dev.

Les appels système effectués sur ces fichiers (open, read, write, close, etc.) sont redirigés par le système d'exploitation vers le driver (pilote) associé au périphérique - un module du noyau.

Il existe deux catégories de device files (fichiers de périphériques) : character device et block device. Les fichiers du type character device correspondent à des périphériques généralement « lents », qui traitent de petites quantités de données et auxquels on n'accède pas souvent. Les opérations sur ces périphériques s'effectuent généralement de manière séquentielle, octet par octet. Ces cas concernent des périphériques tels que les claviers et les souris.

L'objectif de cet exercice sera d'écrire un pilote pour un tel périphérique fictif de type character device, qui stocke stocke simplement tout ce qui lui est écrit.

Les opérations mises en œuvre par un driver sont décrites par la structure struct file_operations (dans include/linux/fs.h). Parmi celles-ci, certaines contiennent des pointeurs vers la mémoire de l'espace utilisateur (user space) qui ne doivent en aucun cas être déréférencés. Il faut plutôt utiliser des fonctions comme - simple_read_from_buffer et simple_write_to_buffer - copy_from_user et copy_to_user.

Consultez le code du noyau (dans include/linux/uaccess.h) et la documentation ci-dessus pour comprendre le fonctionnement de ces fonctions.

Exercice 1

À l'aide de simple_read_from_buffer, simple_write_to_buffer, complétez l'ébauche de code dans le dépôt Git afin d'obtenir un pilote qui :

  • Lors d'un write, prend les données fournies par l'utilisateur et les place dans un buffer du noyau.
  • Lors d'un read, renvoie le contenu du buffer.

Une fois la compilation terminée (utilisez le fichier Makefile fourni), vous pouvez intégrer votre module au noyau à l'aide des commandes suivantes :

insmod echo.ko
mknod /dev/echo c <MAJOR> 0
chmod 666 /dev/mychardev

où <MAJOR> correspond au numéro majeur que vous pouvez obtenir via la commande dmesg. Pour désinstaller votre module, utilisez la commande rmmod et supprimez le fichier de périphérique correspondant.

Exercice 2

Répétez l'exercice ci-dessus en utilisant copy_from_user et copy_to_user. Veillez à bien gérer les offsets, tant pour les lectures que pour les écritures !

Exercices de révision

  • Problème 0 : Supposons qu'un fichier test.txt soit donné, qui contient des caractères ASCII alphabétiques. Définissez un programme qui, en utilisant fork(), lance un processus enfant qui lit le contenu de test.txt à l'exécutable. Le processus parent récupère ensuite le contenu du fichier, le convertit en majuscules, et l'imprime dans un nouveau fichier, nommé test_2.txt.

  • Problème 1 : Créez 3 fichiers nommés f0.txt, f1.txt, f2.txt1 et remplissez-les avec le contenu de votre choix. Créez un programme client-serveur, utilisant TCP/IP, qui permet à un client de demander le contenu de n'importe lequel des trois fichiers au serveur (qui, nous le supposons, a accès à ces trois fichiers).

  • Problème 2 : Développez le programme ci-dessus de manière à ce qu'il puisse gérer plusieurs clients en même temps.

  • Problème 3 : Développez encore plus le programme afin qu'il permette à n'importe quel client de demander une modification de l'un des trois fichiers. Soyez prudents avec les ressources partagées, afin que les demandes simultanées des clients soient traitées de manière atomique.
